// 02 / Perspective
What we hold to in app work.

An app is a product, not a campaign. The first question we ask is whether you need one — because the worst app is the one that exists for marketing reasons and is ignored by the people who own the engineering bill for the next three years.

If the answer is yes, the second question is what you're willing to commit to maintaining. The first version isn't the project. The second and third versions are.

Native, cross-platform, or web. The choice is downstream of two things: how the user is reaching the app today, and what your team can maintain. We've built each. We'll tell you when the platform team you're hiring next is incompatible with the choice you're tempted to make now.

App work is engineering work that happens to ship through an app store. The store is the smaller problem.

// 03 / What we ship
What shipping app work with Modelic looks like.

Native iOS + Android

Swift, SwiftUI, Kotlin. When the experience requires platform fidelity — haptics, gestures, system integrations — native is the answer. We pick it when it's the answer, not when it's familiar.

Cross-platform

React Native, Flutter, where the codebase economics serve a small team that needs to ship to both stores. Honest about the tradeoffs.

Web app

Sometimes "the app" is a web app. PWA, offline-first, push notifications — modern web closes the gap with native for most use cases. We'll tell you when it doesn't.

Integrations + APIs

Backend integrations, auth, push, analytics, in-app purchase, store-listing infrastructure. The plumbing that makes an app feel like a product.

Configurator + product apps

Product configurators, ordering tools, B2B sales apps. Apps that do a specific job extraordinarily well — the kind that gets used daily.

Ongoing app maintenance

OS upgrades, dependency drift, store policy changes, user-reported bugs. We stay in the engagement past launch — apps that don't have a maintenance partner end up ignored.

// 04 / Modern toolkit
Where AI earns its place in app work.
Code generation for repetitive integration work — SDK adapters, schema converters, test scaffolding. LLM features integrated into the product surface only where the use case is real and the failure mode is acceptable. We don't ship a chatbot inside your app because the conference deck said to. The bar for an AI feature in a shipped product is "would this still be useful if the LLM cost ten times more than it does today."
